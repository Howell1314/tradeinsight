### v2.3 主要变更（2026-04-23）

**🔴 关键 Bug 修复：刷新后交易记录丢失**

**问题描述**：用户录入交易后，切换页面或刷新浏览器时，已登记的交易记录完全丢失。

**根因分析**：
- `App.tsx` 的同步 `useEffect` 依赖 `[user?.id]`，但在 `useAuthStore.init()` 完成前，`user` 初始值为 `null`
- useEffect 在 mount 时立刻以 `user=null` 触发，走入 `else` 分支调用 `clearUserData()`
- `clearUserData` 把 `trades: []` 写入 Zustand state，persist 中间件同步覆写 localStorage
- 之后 auth 恢复，`syncFromCloud` 读取本地时已为空，merge 后只剩云端数据
- 若此时云端 upsert 尚未完成（HTTP 请求被页面刷新中断），**该笔交易永久丢失**

**修复方案（commit `b623ca5`）**：
```tsx
useEffect(() => {
  if (!initialized) return     // ← 新增：auth 未就绪时不做任何操作
  if (user) {
    Promise.all([syncFromCloud(user.id), syncJournal(user.id)])...
  } else {
    clearUserData()
    clearJournal()
  }
}, [user?.id, initialized])     // ← 新增 initialized 依赖
```

同时将 `addTrade` 里的 `void upsertTrade(...)` 改为 `.catch(console.error)`，避免网络错误被静默吞没。

**稳定性修复**
- 移除 Yahoo Finance 不可靠的 CORS 代理 fallback（`corsproxy.io` 和 `allorigins.win` 长期不稳定），保留 `query1` → `query2` 两级直连
- 单行查询统一使用 `maybeSingle()` 替代 `single()`，避免"未找到记录"被当作错误抛出
- CSP 配置显式允许 Supabase 域名，修复跨域连接问题

## 1. 项目概述

**TradeInsight** 是一款面向个人投资者的全能交易记录与分析工具，支持多资产类别、多账户管理，并提供专业级的绩效统计、交易日志、K 线图查看功能。

### 核心目标

- 让交易者快速录入、管理所有交易记录，无需依赖 Excel
- 通过 FIFO 自动撮合算法，精确计算每笔盈亏（含手续费）
- 提供多维度统计分析，帮助交易者发现自身盈利模式和失误规律
- 通过交易日志与复盘机制，推动持续改进
- 数据本地优先（LocalStorage），登录后自动云端同步（Supabase）

### 支持的资产类别

| 标识 | 名称 |
|------|------|
| `crypto` | 数字货币 |
| `equity` | 股票 |
| `option` | 期权 |
| `etf` | ETF |
| `cfd` | CFD 差价合约 |
| `futures` | 期货 |

### 数据流向

1. **未登录**：所有数据存储在浏览器 LocalStorage，完全本地离线可用。
2. **登录时**：并行拉取 trades、accounts、account_transactions、risk_rules、journal_entries，与本地数据进行 last-write-wins 合并：
   - 有 `updated_at` 字段的数据：比较时间戳，取较新的一方
   - 本地独有（云端没有）的记录：自动推送至云端
   - 云端独有（本地没有）的记录：拉取到本地
3. **运行时**：每次增删改操作同时写入 LocalStorage（通过 zustand persist）并异步同步至 Supabase（`.catch(console.error)` 暴露错误，v2.3）。
4. **退出登录**：清空内存中的用户数据（trades、journal、prices 等），LocalStorage 恢复为空。

### 路由机制

**不使用 React Router**。通过 `useTradeStore.view` 字符串字段切换页面：

```
'dashboard' | 'trades' | 'positions' | 'analytics' | 'journal' | 'profile' | 'chart'
```

`App.tsx` 的 `pages` 映射表根据 `view` 返回对应组件。

## 4. 数据模型

### 4.1 交易记录（Trade）

```typescript
interface Trade {
  id: string              // 唯一 ID（时间戳 + 随机串）
  account_id: string      // 所属账户 ID
  asset_class: AssetClass // 资产类别
  symbol: string          // 标的代码（如 AAPL、BTC）
  direction: Direction    // buy | sell | short | cover
  quantity: number        // 成交数量（>0）
  price: number           // 成交价格
  total_amount: number    // 成交总额
  commission: number      // 佣金/手续费
  fees: number            // 其他费用（如交易所规费）
  currency: string        // 货币单位（如 USD）
  traded_at: string       // 成交时间（ISO 8601）
  strategy_tags: string[] // 策略标签
  notes: string           // 备注（≤5000 字）
  emotion?: EmotionTag    // 情绪标签
  planned_stop?: number   // 计划止损价（用于计算 R 倍数）
  planned_target?: number // 计划目标价
  metadata: Record<string, unknown> // 扩展元数据（含 contract_multiplier、expiration 等）
  created_at: string      // 创建时间
  updated_at?: string     // 最后修改时间（云端同步冲突解决用）
}
```

**交易方向语义：**

| direction | 含义 | 效果 |
|-----------|------|------|
| `buy` | 买入做多 | 开或加多头仓位 |
| `sell` | 卖出平多 | 减或平多头仓位，触发 FIFO 撮合 |
| `short` | 做空 | 开或加空头仓位 |
| `cover` | 回补（平空）| 减或平空头仓位，触发 FIFO 撮合 |

interface Account {

### 4.3 已结算交易（ClosedTrade）

由 FIFO 引擎自动计算，**非用户直接录入**：

```typescript
interface ClosedTrade {
  id: string              // 确定性 ID：ct__<开仓tradeId>__<平仓tradeId>__<数量>
  symbol: string
  asset_class: AssetClass
  account_id: string
  direction: 'long' | 'short'
  open_trades: Trade[]    // 构成开仓的原始交易
  close_trades: Trade[]   // 构成平仓的原始交易
  open_price: number
  close_price: number
  quantity: number
  gross_pnl: number       // 毛利润
  net_pnl: number         // 净利润（扣除手续费和其他费用）
  commission: number
  fees: number
  opened_at: string
  closed_at: string
  holding_days: number
  strategy_tags: string[]
  initial_risk?: number   // 初始风险金额（有止损价时计算）
  actual_r?: number       // 实际 R 倍数 = net_pnl / initial_risk
}
```

## 5. 功能模块详解

### 5.1 仪表盘（Dashboard）

核心概览页，展示所有关键指标和图表。

**统计指标（第一行）：** 总盈亏、胜率、盈亏比、最大回撤
**统计指标（第二行）：** 期望值、利润因子、夏普比率、平均持仓、总回报率

**图表：** 累计收益曲线、品种盈亏分布、按星期分析、交易统计快览

**风险规则预警：** 触发日/周亏损限制时红色警告；达成月度目标时绿色提示。

---

### 5.2 交易记录（Trades）

原始交易流水管理页，所有操作均会触发重新计算持仓和统计。

**功能：** 添加/编辑/删除交易、卖出/回补数量校验、批量 CSV 导入/导出。

**筛选条件：** 关键词、资产类别、方向、月份、自定义日期范围。

**分页：** 每页 20 条。

---

### 5.3 持仓管理（Positions）

显示当前所有未平仓持仓，数据由 FIFO 引擎自动维护。

**功能：**
- 多头/空头持仓分开展示
- **一键刷新行情**：Yahoo Finance 两级 fallback（`query1` → `query2`，v2.3 精简）
- 可手动输入当前市价，实时计算浮动盈亏
- 合约乘数自动纳入盈亏计算
- **价格过期提示**：>30 分钟显示灰色"X 分钟前"，>4 小时显示橙色⚠ 警告

---

### 5.4 统计分析（Analytics）

五标签深度分析页：综合分析 / 标的分析 / 时间分析 / 策略分析 / 风险指标。

**风险指标标签**：回撤曲线、R 倍数分布、情绪与盈亏关联、失误标签成本（响应式读取 `useJournalStore`）。

---

### 5.5 交易日志（Journal）

结构化交易日记，帮助交易者养成复盘习惯。

**字段：** 日期、当日盈亏、情绪、四段式总结（已做好 / 需改进 / 明日计划 / 总结）、失误标签多选、截图 ≤5 张、复盘笔记。

**联动：** 查看日志详情时自动展示该日期的已结算交易。

**移动端**：列表/详情两栏布局在移动端切换为全屏单栏。

---

### 5.6 K 线图（Chart，v2.2 新增）

独立的 K 线图页面，便于快速查看标的走势。

**技术实现：**
- 组件：`components/chart/TradingViewWidget.tsx`，加载 TradingView 官方 Widget
- 主题管理：`useChartStore`（Zustand + persist，localStorage key = `tradeinsight:chart`）
- 默认暗色主题，可切换

**数据源：** 通过 Supabase Edge Function `fetch-ohlc` 从 OKX `/api/v5/market/candles` 拉取（v2.2 由 Binance 切换到 OKX，国内访问更稳定）。

**backtest 储备**：`algo/` 目录和 `run-backtest` Edge Function 为后续"策略回测"功能预留底座，当前未在 UI 暴露。

### 5.7 个人资料（Profile）

账户配置和风险规则设置页。

**功能：** 修改昵称/头像、账户 CRUD、出入金记录（已云同步）、风控规则设置（已云同步）。

## 7. 核心计算逻辑

### 7.1 FIFO 撮合算法（`processTradesIntoPositions`）

所有交易按时间排序后，按标的 + 账户分组，分别维护多头队列（`longQueue`）和空头队列（`shortQueue`）：

```
买入（buy）：
  1. 优先撮合空头队列（平空）→ 生成 ClosedTrade（short）
  2. 剩余数量加入多头队列

卖出（sell）：
  1. 优先撮合多头队列（平多）→ 生成 ClosedTrade（long）
  2. 剩余数量加入空头队列（开新空仓）

做空（short）→ 直接加入空头队列
回补（cover）→ 优先撮合空头队列，剩余加入多头队列
```

盈亏计算：
- 多头：`gross_pnl = (close_price - open_price) × qty × multiplier`
- 空头：`gross_pnl = (open_price - close_price) × qty × multiplier`
- `net_pnl = gross_pnl - 按比例分摊的开平仓（commission + fees）`

分批建仓均价：`avg_cost = Σ(price × remaining) / Σremaining`（数量加权）

### 7.2 derive() 缓存机制

```typescript
let _coreCache: { trades; selectedAccount; closedTrades; rawPositions; stats } | null = null

function deriveCore(trades, selectedAccount) {
  if (_coreCache?.trades === trades && _coreCache.selectedAccount === selectedAccount)
    return _coreCache
  // 否则全量重算并缓存
}

function derive(trades, selectedAccount, currentPrices) {
  const { closedTrades, rawPositions, stats } = deriveCore(trades, selectedAccount)
  return { closedTrades, openPositions: applyCurrentPrices(rawPositions, currentPrices), stats }
}
```

价格刷新（`updateCurrentPrice`）不经过 `derive`，直接调用 `applyCurrentPrices`，完全绕过重算。

### 7.3 统计指标计算（`computeStats`）

| 指标 | 计算公式 |
|------|---------|
| 胜率 | `盈利笔数 / 总笔数` |
| 盈亏比 | `平均盈利 / 平均亏损` |
| 期望值 | `胜率 × 平均盈利 - 败率 × 平均亏损` |
| 最大回撤 | 逐笔累加净值，记录峰值，`max((峰值 - 当前净值) / 峰值)` |
| 利润因子 | `总盈利 / 总亏损` |
| 夏普比率 | 先按交易日聚合 PnL → 日均值/日标准差 × √252 年化 |
| 连胜/连亏 | 遍历时序记录连续次数峰值 |

### 7.4 浮动盈亏

```
多头：unrealized_pnl = (current_price - avg_cost) × |qty| × multiplier
空头：unrealized_pnl = (avg_cost - current_price) × |qty| × multiplier
```

### 7.5 R 倍数

```
initial_risk（多头）= (open_price - planned_stop) × qty × multiplier
initial_risk（空头）= (planned_stop - open_price) × qty × multiplier
actual_r = net_pnl / initial_risk
```

## 8. 关键架构约束（Do's & Don'ts）

### ⚠️ Zustand persist 与 auth 初始化顺序（v2.3 固化）

**背景**：`useTradeStore` 和 `useJournalStore` 使用 `persist` 中间件，**任何 `set()` 都会同步覆写 localStorage**。在 auth 初始化完成前，`user` 初始值为 `null`，此时若误调 `clearUserData()`，会把空数组写入 localStorage，导致尚未 upsert 到云端的数据永久丢失。

**规则**：任何基于 `user` 状态的清理副作用，必须先判断 `useAuthStore.initialized === true`。

**反例（v2.3 修复前）：**
```tsx
useEffect(() => {
  if (user) { sync(user.id) } else { clearUserData() }
}, [user?.id])  // ❌ auth 未就绪时也会触发 clearUserData
```

**正例：**
```tsx
useEffect(() => {
  if (!initialized) return            // ✅ 等 auth 就绪
  if (user) { sync(user.id) } else { clearUserData() }
}, [user?.id, initialized])
```

### 🔇 异步写入错误必须显式捕获

- ❌ `void upsertTrade(userId, trade)` —— 错误被静默吞没
- ✅ `upsertTrade(userId, trade).catch(e => console.error('[sync] ...', e))`

### 🔑 ID 必须稳定

- `ClosedTrade.id` 和 `Position.id` 采用确定性生成（基于构成 trade 的 ID），不能用 `Math.random` 或 `Date.now()`，否则会导致 React key 漂移引发重渲染和动画抖动。

### 📍 查询单行使用 maybeSingle()

- Supabase 查询可选单行（如 `profiles`、`user_settings`）统一使用 `maybeSingle()`，避免"未找到"被当作错误。

## 9. 安全机制

### 输入验证

所有从 LocalStorage 恢复的数据在使用前通过 `isValidTrade` / `isValidAccount` 校验：

- 字段类型校验（string / number / array）
- 数值范围校验（price ≥ 0，quantity > 0，isFinite 检测）
- 字符串长度上限（symbol ≤ 30，notes ≤ 5001，id ≤ 64）
- 枚举白名单校验（asset_class、direction 必须在允许集合内）
- metadata 必须为非 null 的普通对象（防止原型污染）

日志条目通过 `isValidEntry` 校验：
- 日期格式：`/^\d{4}-\d{2}-\d{2}$/`
- 情绪标签白名单
- 图片 URL 数量 ≤ 10

### 业务规则校验

- `validateNewTrade`：卖出/回补前检查当前持仓是否充足，防止超卖/超平

### 文件上传限制

- 日志截图单张 ≤ 10MB，仅接受 `image/*` MIME 类型
- 每条日志最多 5 张图片

### Supabase 安全

- 所有数据库操作均通过 Row Level Security（RLS）绑定 `user_id`
- 客户端使用 `@supabase/supabase-js` 官方 SDK，不直接拼接 SQL
- CSP 策略白名单仅包含自有 Supabase 域名（v2.3 明确）
